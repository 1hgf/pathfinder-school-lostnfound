## 差异清单

### 1. 地图实现方式根本性差异

**分类**: 核心架构  
**严重程度**: 🔴 高

**设计文档**:
- 使用高德地图 JS API 作为地图底图
- 在高德地图上标注建筑物
- 使用高德地图的经纬度坐标系统

**实际实现**:
- 使用 Canvas 绘制本地图片 (`mapFX_med.jpg`)
- 完全自绘地图，无第三方地图服务
- 使用像素坐标系统 `[x, y]`

**影响**:
- 实现方式完全不同
- 不依赖高德地图API key和网络连接
- 更灵活但失去了高德地图的底图更新和标准化
- 前端技术栈从"Vue 3 + 高德地图"变为"原生Canvas"

**建议**:
- [ ] 保持设计文档，按原计划使用高德地图API
- [ ] 修改设计文档，改为Canvas方案（当前实现）
- [ ] 混合方案：提供两种地图模式供用户选择

【审阅意见】: tumu1t: 维持 Canvas 方案，弃用高德地图API方案。

### 2. 坐标系统差异

**严重程度**: 🔴 高

**设计文档**:

```sql
`lng` DOUBLE NOT NULL COMMENT '经度'
`lat` DOUBLE NOT NULL COMMENT '纬度'
```

**实际实现**:
```javascript
buildingData = [[665.7, 281.3], [577.1, 255.6], ...]

```
像素坐标，无地理意义

**影响**:
- 数据库表结构需要调整
- 前后端接口的坐标格式不一致
- 无法与真实地理位置关联

**建议**:
- [ ] 数据库改为存储像素坐标 `x, y`
- [ ] 或者实现坐标转换层（像素 ↔ 经纬度）
- [ ] 在设计文档中明确坐标系统选择

【审阅意见】: 坐标改用png坐标。


### 3. 导航模式差异

**分类**: 功能范围  
**严重程度**: 🟡 中

**设计文档**:
- 只提到步行导航
- `walk_time` 字段（步行时间）
- 步行速度: 1.2 m/s

**实际实现**:
```javascript
function planNavigation(start, end, mode) {
    // mode: 'bike' 或 'walk'
}
```
- 支持骑行(bike)和步行(walk)两种模式
- 骑行只走 `roadData`（车行道）
- 步行可走 `roadData + pathData`（车行道+人行道）

**影响**:
- 设计文档遗漏了骑行模式
- 需求分析中未体现这一功能
- 数据库设计未考虑路径类型区分

**建议**:
- [ ] 设计文档补充骑行导航需求
- [ ] 数据库 `map_edge` 表增加 `vehicle_type` 字段区分
- [ ] 或简化实现，去掉骑行模式只保留步行

【审阅意见】: 加上路径类型。

### 4. 路径数据结构差异

**分类**: 数据模型  
**严重程度**: 🟡 中

**设计文档**:
- 只有一种边：`map_edge` 表
- `edge_type`: OUTDOOR/INDOOR/STAIRS/ELEVATOR

**实际实现**:
```javascript
let roadData = [[67,8], [50,101], ...];  // 车行道（红色）
let pathData = [[102,101], [100,101], ...];  // 人行道/捷径（黄色）
```
- 两类边数据分开存储
- `roadData`: 车辆可通行，骑行和步行都可走
- `pathData`: 仅步行可走的捷径

**影响**:
- 数据库需要区分路径类型
- 寻路算法需要根据交通方式筛选可用边

**建议**:
- [ ] 在 `map_edge` 表增加 `accessible_by` 字段 ('WALK'/'BIKE'/'BOTH')
- [ ] 或者在 `edge_type` 枚举中增加 'ROAD'/'PATH'
- [ ] 更新接口文档，路径规划API增加 `mode` 参数

【审阅意见】: 导航做类型区分。

**影响**:

- 缺少建筑类型图标区分

**建议**:
- [ ] 隐藏 crossing 节点的显示（仅用于寻路计算）
- [ ] 实现建筑类型的图标或颜色区分
- [ ] 或简化设计文档，承认当前简单实现

【审阅意见】: 地点做颜色/图标Emoji区分即可，不必要显示出来，但这个仍然需要在数据库反馈。

### 8. 寻路算法实现细节

**分类**: 算法  
**严重程度**: 🟢 低

**设计文档**:
- 描述了 Dijkstra 算法伪代码
- 使用优先队列
- 基于距离的权重

**实际实现**:
```javascript
// 简单数组实现的优先队列
const nodes = Array.from(edgeMap.keys());
while (nodes.length) {
    let bestI = 0;
    for (let i = 1; i < nodes.length; i++) {
        if (dist.get(nodes[i]) < dist.get(nodes[bestI])) bestI = i;
    }
    const u = nodes[bestI];
    nodes.splice(bestI, 1);
    // ...
}
```
- 未使用真正的优先队列（性能略差）
- 但对于校园规模（<200节点）完全够用

**影响**:
- 算法复杂度理论上 O(V²) 而非 O((V+E)logV)
- 实际性能在当前规模下无影响

**建议**:
- [ ] 保持当前实现（简单且够用）
- [ ] 或者使用堆实现真正的优先队列（优化性能）
- [ ] 设计文档可以说明"简化实现"

【审阅意见】: V^2 无所谓，那个实现错了得改掉。

### 13. 距离和时间计算

**分类**: 算法细节  
**严重程度**: 🟢 低

**设计文档**:
```
步行速度：1.2 m/s
duration(秒) = distance(米) / 1.2
```

**实际实现**:
```javascript
// 距离使用平方（避免开方运算）
function navDist2(pt1, pt2) {
    const dx = pt1[0] - pt2[0], dy = pt1[1] - pt2[1];
    return dx * dx + dy * dy;  // 返回距离平方
}
```
- 未转换为实际米数
- 未计算步行时间文本

**影响**:
- 用户看不到"约6分钟"这样的提示
- 像素距离无现实意义

**建议**:
- [ ] 增加像素到实际距离的转换系数
- [ ] 计算并显示预计步行/骑行时间
- [ ] 在设计文档中说明转换方式

【审阅意见】: 支持

## 总结建议

### 方案A：修订设计文档以匹配实现

如果当前Canvas实现满足需求，建议修订设计文档：

1. ✅ 将地图方案从"高德地图API"改为"Canvas + 本地图片"
2. ✅ 坐标系统改为像素坐标
3. ✅ 补充骑行/步行两种导航模式
4. ✅ 补充路径类型区分（roadData / pathData）
5. ✅ 补充精美的路径绘制UI效果描述
6. ✅ 说明节点索引约定（负数=建筑，正数=路口）

### 方案B：按设计文档重构实现

如果要严格按设计文档开发：

1. ❌ 去掉当前HTML，用Vue 3重写前端
2. ❌ 集成高德地图JS API
3. ❌ 实现后端API，数据存储到MySQL
4. ❌ 使用经纬度坐标系统
5. ❌ 实现完整的前后端分离架构

### 方案C：混合方案（推荐）

将当前HTML作为地图组件嵌入Vue项目：

1. ✅ 主系统按设计文档用Vue开发
2. ✅ 地图页面嵌入当前Canvas实现（iframe或组件）
3. ✅ 地图数据迁移到数据库，通过API加载
4. ✅ 保留Canvas的精美绘制效果
5. ✅ 坐标系统统一为像素坐标
